# Claude 开发说明书：语音输入法产品

你现在扮演一名资深全栈工程师、产品工程负责人和 AI 应用架构师。请根据以下需求，直接开始设计并实现一款“智能语音输入法”产品。目标是做出一个可运行的 MVP，并为后续迭代预留扩展空间。

---

## 1. 项目目标

开发一款面向 **办公、聊天、创作** 三类场景的语音输入法产品。产品不只是“语音转文字”，而是要在识别后进行场景化处理，包括：

- 专业词汇 / 英文名 / 人名 / 品牌名的准确识别与纠错
- 基于用户词库和个人记忆的自学习机制
- 办公场景下的正式化润色与多版本改写
- 聊天场景下的轻量、自然、带情绪表达的文本推荐
- 创作场景下的灵感记录、整理、分段、提纲化输出
- 保留原文，允许用户选择是否使用 AI 优化结果

核心目标是平衡以下四点：

1. **准确度**：尽量减少专业词识别错误
2. **易用性**：操作简单，用户能快速上手
3. **响应速度**：先出原文，再异步做优化
4. **成本**：高频能力尽量轻量化，重能力按需调用

---

## 2. 产品定位

这是一个“场景化语音输入法”，不是单纯的语音识别工具。

### 2.1 场景定义

**办公模式**
- 适合会议纪要、邮件、文档、汇报、项目沟通
- 重点：专业术语识别、正式表达、结构化整理

**聊天模式**
- 适合微信、QQ、短信、社交平台等输入场景
- 重点：快速转写、自然口语、标点、emoji、颜文字

**创作模式**
- 适合灵感记录、小说大纲、口播稿、自媒体文案
- 重点：保留原意、整理结构、扩写、提炼、回溯历史

请优先实现一个可交付的最小可用版本，范围如下：

### 3.1 必做功能

1. 基础语音转文字
2. 实时展示原文
3. 自动标点
4. 场景切换：办公 / 聊天 / 创作
5. 用户词库管理
6. 导入行业词库或个人文档提取词条
7. 识别不确定词时高亮提示
8. 用户确认替换后写入记忆
9. 办公模式下提供润色建议
10. 聊天模式下提供 emoji / 颜文字建议
11. 创作模式下提供分段、提纲、标题建议
12. 历史记录保存与检索

### 3.2 暂缓功能

以下功能可以作为第二阶段，不要求首版全部完成：

- 更复杂的情绪分级打分系统
- 多端同步
- 深度个性化风格学习
- 高级会员体系
- 离线端到端本地大模型

## 4. 关键产品原则

实现时必须遵守以下原则：

1. **原文优先**：先给用户原始转写结果，不直接强行替换
2. **可选优化**：AI 推荐必须让用户可以选择，不可强制改写
3. **低延迟优先**：实时转写要快，优化可以异步完成
4. **记忆可控**：用户的词库和修正记录可查看、可删除、可关闭
5. **场景分层**：办公、聊天、创作使用不同策略
6. **逐步学习**：用户每次修正都应进入学习闭环

## 5. 用户需求拆解

### 5.1 办公用户需求

痛点：
- 专业词汇、缩写、英文名称容易错
- 文本过于口语化，不够正式
- 会议纪要、邮件、汇报需要结构化

产品能力：
- 支持行业选择，如互联网、金融、法律、医疗、教育、游戏等
- 支持导入文档、通讯录、项目资料
- 自动提取高频名词加入个人词库
- 对低置信度专有名词做波浪线提示
- 提供严谨版、商务版、简洁版改写

### 5.2 聊天用户需求

痛点：
- 输入要快
- 标点和语气要自然
- 想要更有情绪、更有亲和力的表达

产品能力：
- 快速转写，优先响应速度
- 自动补标点
- 识别语义和情绪
- 推荐 emoji、颜文字、轻量口语化改写
- 支持娱乐模式开关

### 5.3 创作用户需求

痛点：
- 灵感零散，容易丢失
- 说出来的话杂乱
- 想要快速整理成可复用内容

产品能力：
- 保留原始语音内容
- 自动分段
- 提炼提纲
- 支持标题、摘要、扩写、润色
- 支持历史回看和版本管理

## 7. 产品交互要求

### 7.1 语音输入主流程

1. 用户点击开始录音
2. 系统实时转写为文本
3. 页面先显示原文
4. 对低置信度词汇做标记
5. 用户可点击标记词查看候选词
6. 用户选择后替换
7. 系统记录修正结果
8. 根据场景可选择是否生成优化版本

### 7.2 办公模式交互

- 显示：原文 + 优化建议
- 提供 2~3 版风格建议
- 每版需说明适用场景
- 不直接覆盖原文，默认保留原始内容

### 7.3 聊天模式交互

- 显示：原文 + 推荐表情
- 可推荐两版风格不同的 emoji / 颜文字组合
- 标点尽量自然，不要过于“机器感”
- 支持一键插入到输入框

### 7.4 创作模式交互

- 显示：原文
- 提供整理版、提纲版、扩写版
- 支持一键生成标题、摘要、分点
- 保存到历史记录，方便回溯

## 10. 逻辑规则

### 10.1 专有名词识别规则

当语音识别结果中出现疑似专业词、人名、英文名称时：
1. 优先检查用户词库
2. 再检查行业词库
3. 再根据上下文语义判断
4. 低置信度时不强改，改为高亮提示
5. 用户确认后写入记忆

### 10.2 优化规则

- 办公场景：优先正式、简洁、条理化
- 聊天场景：优先自然、轻松、情绪化
- 创作场景：优先保留原意，再辅助结构化

### 10.3 表情推荐规则

- 先判断情绪
- 再判断语境
- 再判断用户历史偏好
- 输出 2 个备选方案，不要只给一个

### 10.4 学习规则

用户每次完成以下操作，系统都应记录：
- 替换词语
- 选择某个润色版本
- 采纳某个 emoji / 颜文字
- 删除或修改 AI 输出

这些行为应作为后续推荐的重要信号。

## 16. 额外说明

这个产品最重要的不是“语音识别本身”，而是“识别后如何更聪明地帮用户表达”。

所以实现时请特别注意：
- 不要牺牲速度去追求不必要的复杂度
- 不要强行替换用户文本
- 不要让 AI 输出看起来太机械
- 不要让用户必须手动做太多选择
- 不要让学习机制污染用户词库

请以“先能用，再变聪明”为原则推进。
